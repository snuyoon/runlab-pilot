# RunLab 앱 프로젝트 계획서
## AI 스마트 러닝워치 연구 참여자 컴플라이언스 관리 앱

> ⚠️ **역사 자료**: VC 데모 버전(2026-04) 시절 계획서입니다. 현재 파일럿 시스템과 다릅니다 — 최신 상태는 [DEVLOG.md](DEVLOG.md)와 [ARCHITECTURE.md](ARCHITECTURE.md)를 보세요.

---

## 1. 프로젝트 개요

### 목적
- 연구 참여자의 수면 시계 착용률(컴플라이언스) 극대화
- 기상 직후 EMA(Ecological Momentary Assessment) 수집
- 게이미피케이션을 통한 참여 동기 부여
- 연구팀용 대시보드를 통한 참여자 모니터링

### 사용자
- 연구 참여자 (러너 160명)
- 연구 관리자 (성재님, 교수님)

### 핵심 기능
1. 취침 리마인더 + 워치 착용 안내
2. 기상 알람 + EMA 설문 (10초 이내 완료)
3. 캐릭터 성장 시스템 (컴플라이언스 연동)
4. 관리자 대시보드 (착용률, 응답률 모니터링)

---

## 2. 기술 스택

| 영역 | 기술 | 선택 이유 |
|---|---|---|
| 프론트엔드 | React Native (Expo) | iOS/Android 동시 지원, Claude Code로 빠른 개발 |
| 백엔드 | Supabase | 인증, DB, 실시간 구독 올인원, 무료 티어 충분 |
| 알림 | Expo Notifications | 로컬 알람 + 푸시 알림 |
| 배포 | EAS Build | Expo 빌드 서비스로 앱스토어 배포 |
| 관리자 대시보드 | Next.js + Vercel | 웹 기반, 빠른 구축 |

### 왜 React Native(Expo)?
- Flutter 대비 Claude Code와의 호환성이 높음
- Expo로 네이티브 알람 기능 지원
- 앱스토어 배포까지 EAS로 원스톱

## 3. 개발 단계

### Phase 1: 기반 구축 (1주차)

**목표**: 프로젝트 세팅 + 인증 + DB 스키마

```
작업 목록:
├── Expo 프로젝트 초기화
├── Supabase 프로젝트 생성
├── DB 스키마 설계 및 생성
│   ├── users (참여자 정보, 그룹, 연구ID)
│   ├── sleep_logs (날짜, 착용여부, 수면시작/종료)
│   ├── ema_responses (수면만족도, 근육통, 기분, 타임스탬프)
│   ├── running_logs (운동일자, 연동 데이터)
│   ├── character_status (레벨, 경험치, 연속일수)
│   └── compliance_summary (주간/월간 착용률)
├── 회원가입/로그인 (연구 참여 코드 기반)
└── 기본 네비게이션 구조
```

**주의사항**:
- 회원가입은 이메일이 아닌 **연구 참여 코드** 방식으로 (개인정보 최소화, IRB 대응)
- 참여자 ID는 비식별화 처리 (SNU-001 형태)
- Supabase RLS(Row Level Security) 반드시 설정

---

### Phase 2: 핵심 기능 - 알람 + EMA (2주차)

**목표**: 취침 리마인더 → 기상 알람 → EMA 설문 플로우 완성

```
작업 목록:
├── 취침 리마인더
│   ├── 사용자가 설정한 시간에 로컬 알림
│   ├── 알림 탭 → "워치를 착용하고 주무세요!" 화면
│   └── 착용 확인 버튼 (선택사항, 안 눌러도 무방)
├── 기상 알람
│   ├── 네이티브 알람 기능 (소리, 진동)
│   ├── 알람 해제 시 자동으로 EMA 화면 전환
│   └── 스누즈 기능
└── EMA 설문 화면
    ├── Q1: "오늘 수면은 어떠셨나요?" (5점 이모지 척도)
    ├── Q2: "근육통은 어떠세요?" (5점, 전날 운동 기록 있을 때만)
    ├── Q3: "오늘 기분은 어떠세요?" (5점 이모지 척도)
    ├── 한 화면에 3문항, 탭 한번씩 = 3초
    └── 제출 → 캐릭터 반응 애니메이션 → 홈으로
```

**주의사항 - 척도 설계 (IRB/논문 대응)**:

| 문항 | 척도 | 근거 |
|---|---|---|
| 수면 만족도 | 5점 Likert (1=매우 나쁨 ~ 5=매우 좋음) | PSQI 단일문항 변형 |
| 근육통(DOMS) | 5점 (1=없음 ~ 5=매우 심함) | VAS 간소화 |
| 기분 | 5점 (1=매우 나쁨 ~ 5=매우 좋음) | Feeling Scale 단일문항 |

- 이모지를 사용하되, 데이터 저장은 반드시 숫자(1~5)로
- 설문 완료 타임스탬프 자동 기록 (기상 시각 추정에 활용)
- EMA 미응답 시 2시간 후 리마인더 푸시 1회 발송, 그 이후 미응답은 NULL 처리

### Phase 3: 캐릭터 시스템 (3주차)

**목표**: 게이미피케이션 캐릭터 성장 시스템

```
작업 목록:
├── 캐릭터 디자인
│   ├── 기본 캐릭터 (귀여운 러닝 마스코트)
│   ├── 레벨별 외형 변화 (Lv.1~10, 10단계)
│   ├── 상태: 활발 / 졸림 / 깊은잠 (컴플라이언스에 따라)
│   └── Lottie 애니메이션으로 구현
├── 경험치 시스템
│   ├── EMA 응답 완료: +10 XP
│   ├── 워치 착용 수면 데이터 확인: +20 XP
│   ├── 러닝 데이터 업로드: +15 XP
│   ├── 연속 7일 달성: +50 XP 보너스
│   └── 레벨업 기준: Lv.1=0, Lv.2=100, Lv.3=250 ... (점진적 증가)
├── 페널티 시스템
│   ├── 1일 미응답: 캐릭터 "졸림" 상태
│   ├── 3일 연속 미응답: 캐릭터 "깊은 잠" 상태 + XP -30
│   ├── 7일 연속 미응답: 레벨 1단계 하락
│   └── "죽음" 표현 사용 금지 (IRB 고려)
└── 홈 화면
    ├── 캐릭터 + 현재 레벨/XP 바
    ├── 오늘의 미션 체크리스트
    ├── 연속 일수 카운터
    └── 이번 주 착용률 요약
```

**주의사항**:
- 캐릭터 에셋은 **Lottie 애니메이션** 사용 (가볍고 부드러움)
- 초기에는 단순한 SVG 캐릭터로 시작, 이후 디자이너 투입 가능
- 캐릭터 상태 변화가 "심리적 압박"이 아닌 "귀여운 동기부여"가 되도록 톤 조절
- 레벨 데이터는 연구 종료 시 보너스 사례금 산정 기준으로 활용 가능 (IRB에 명시)

### Phase 4: 데이터 연동 + 러닝 기록 (4주차)

**목표**: 가민/브이씨 워치 데이터 연동, 러닝 기록 관리

```
작업 목록:
├── 가민 커넥트 API 연동
│   ├── OAuth2 인증
│   ├── 수면 데이터 자동 수집 (수면시간, 단계, 심박, HRV)
│   ├── 러닝 데이터 자동 수집 (거리, 페이스, 심박, 케이던스)
│   └── 일일 자동 동기화 (백그라운드)
├── 수동 데이터 업로드 (가민 미사용자 대비)
│   ├── FIT/GPX 파일 업로드
│   └── 수동 입력 폼 (최소한의 필드)
├── 러닝 기록 화면
│   ├── 캘린더 뷰 (운동일/수면일 표시)
│   ├── 러닝 요약 (주간 마일리지, 페이스 추이)
│   └── 수면 요약 (주간 평균 수면시간, 만족도 추이)
└── 데이터 검증
    ├── 수면 데이터 유효성 체크 (3시간 미만 → 경고)
    ├── 러닝 데이터 이상치 필터링
    └── 누락 데이터 알림
```

**주의사항**:
- 가민 API는 Health API (developer.garmin.com) 사용
- API 승인에 2~4주 소요될 수 있으므로 Phase 1에서 미리 신청
- 브이씨 워치 SDK는 브이씨 측에서 제공받아야 함 → 카이스트/브이씨와 조율 필요
- 가민 API 승인 전까지는 수동 업로드로 대체

**목표**: 연구팀이 참여자 현황을 한눈에 파악

```
작업 목록:
├── Next.js 웹 대시보드
├── 참여자 목록
│   ├── ID, 그룹(성별/연령/숙련도), 참여 시작일

│   └── 캐릭터 레벨
├── 전체 현황 요약
│   ├── 그룹별 평균 착용률 차트
│   ├── 일별 응답률 추이
│   ├── 탈락 위험 참여자 경고 (착용률 50% 미만)
│   └── 데이터 수집 진행률
├── 데이터 내보내기
│   ├── CSV 일괄 다운로드 (EMA, 수면, 러닝)
│   ├── 날짜 범위 필터
│   └── 참여자별 / 그룹별 필터
└── 알림 관리
    ├── 특정 참여자에게 푸시 알림 전송
    └── 전체 공지 발송
```

## 6. IRB 관련 체크리스트

앱 개발 시 IRB 심의에 필요한 내용을 미리 준비:

- [ ] 수집 데이터 목록 명시 (EMA 3문항, 수면 센서 데이터, 러닝 데이터)
- [ ] 데이터 저장 위치 및 암호화 방식 (Supabase, 서울대 IRB 기준)
- [ ] 개인정보 비식별화 방안 (연구ID만 사용, 이름/전화번호 미저장)
- [ ] 캐릭터 시스템이 심리적 압박을 주지 않음을 소명
- [ ] 사례금 차등 지급 시 기준 명시 (착용률 기반, 캐릭터 레벨 참고)
- [ ] 앱 사용 동의서 문구 (연구 참여 동의서에 포함)
- [ ] 데이터 보관 기간 및 파기 계획

## 8. Claude Code 개발 전략

### 프로젝트 구조
```
runlab/
├── apps/
│   ├── mobile/          # React Native (Expo) 앱
│   └── dashboard/       # Next.js 관리자 대시보드
├── packages/
│   └── shared/          # 공통 타입, 유틸리티
└── supabase/
    ├── migrations/      # DB 마이그레이션
    └── functions/       # Edge Functions
```

### Claude Code 활용 팁
1. Phase별로 CLAUDE.md에 현재 단계 컨텍스트 기록
2. 각 Phase 시작 시 해당 Phase의 요구사항을 CLAUDE.md에 명시
3. Supabase 스키마는 SQL 마이그레이션 파일로 버전 관리
4. 컴포넌트 단위로 작업 지시 (한번에 전체 앱 생성 X)
5. 테스트 코드 병행 작성 (EMA 데이터 정합성 검증)
